# Capture Project Memory

## 目标

从当前开发过程里提炼真正值得长期保留的信息。不要把一次性背景、临时问题、普通小错写进长期文档。

## 流程

1. 收集证据
   - 查看本次涉及的文件、diff、命令输出、用户明确偏好
   - 读取已有 AGENTS.md、相关 skill、postmortem 索引，避免重复记录

2. 判断是否值得记录
   - 是否会在未来重复出现
   - 是否忽略后会造成返工、错误实现或架构偏离
   - 是否能写成明确的行为规则或可复用流程
   - 是否基于当前上下文中的事实证据

3. 选择记录位置
   - 项目长期约定、模块硬规则 → AGENTS.md
   - 可复用工作流 → skill
   - 重大事故、重复错误、高影响 bug → postmortem
   - 不满足长期价值 → 不记录

## AGENTS.md

- 只记录重要、非一次性的项目约定和约束
- ** 必须是项目相关的, 不要因为其他原因的错误也记录进项目中 **
- 按就近原则选择涉及范围内最近的已存在 AGENTS.md；否则使用项目根目录 AGENTS.md
- 使用二级标题分组，标题下通常用无序列表
- 不记录一次性任务背景、临时路径、普通 debug 过程

## Skill

- ** 必须是项目强相关的 **
- 写 skill 前先读取 `write-a-skill`, 缺失时提醒且不做记录
- skill 记录可复用流程，不记录单次错误; 重要、非一次性的
- 只有当当前上下文中存在未来肯定会重复执行的流程时，才新增或更新 skill
- 语言精简凝练，模仿当前项目已有 skill 风格
- 如果有踩坑点也应该考虑是否以后会重复且重要, 如果一次解决不应该记录在skill

## Postmortem

- 写尸检前先读取 `postmortem`, 缺失时提醒且不做记录
- 小修小补不要记录
- 重点记录重大错误、重复错误、高影响事故
- 用户未主动要求时，只有在影响范围明显较大时才建议记录，不硬写

## 例外

- subagent 不要做任何操作, 仅在任务完成的最后提供建议(如果有)

## 格式

**不管你想写入什么文档, 必须遵循格式要求**

### 标点规则

- 禁止用破折号（—），用逗号或分号替代
- 使用「」而不是“”
