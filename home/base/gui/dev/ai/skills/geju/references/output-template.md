# Geju 输出骨架

当用户需要打开设计空间、挑战保守局部思维时，使用此骨架。

```markdown
# 格局判断：<topic>

## Thesis
<尖锐、有杠杆的假设，1-3 句。不要当作定论呈现。>

## Confidence
- **信心等级**: high / medium / low
- **为什么不确定**: <缺失的证据、未解决的约束、或依赖的假设>

## 陷阱
- **继承来的约束**: <兼容性 / 迁移 / 旧命名 / 局部细节 / 半成品实现 / 模糊目标>
- **它真实吗？**: yes / no / partially
- **为什么**: <具体的契约名字，或说明为什么只是惯性>

## 高格局方向
<如果不被当前实现过度约束，干净的目标模型是什么。>

## 打开格局的打法
- **使用的打法**: 从终局倒推 / 零历史包袱假设 / 杀掉错误概念 / 十倍问题 / 反向约束 / 不可妥协原则 / 有品位地删除
- **它揭示了什么**: <更高层面的洞察>

## Bold Takes
- <有依据但大胆的主张>
- <什么应该被删除、合并、拆分、改名或重塑>
- <什么不该仅因为它已存在就保留>

## 方案选项
| 方案 | 优化目标 | 代价 | 结论 |
|------|----------|------|------|
| 保守路线 | <兼容性/局部安全> | <复杂度代价> | 否决 / 仅当…才保留 |
| 干净目标 | <长期清晰度> | <迁移/重构代价> | 推荐 / 不推荐 |
| 分阶段抵达干净目标 | <平衡路线> | <额外的工作量> | 推荐 / 兜底方案 |

## 不要做的事
- <要避免的局部优化、兼容 shim、部分修补、细节陷阱>
- <表面安全但保留了错误模型的事情>

## 第一个证明点
<能证明这个方向的最小产物——测试、文档改动、trace、原型、或决策。>

## 证伪条件
<什么证据会证明这个假设是错的，或迫使你转向不同的方向。>
```
