# 核心行为

在开始写任何代码之前，先阅读可用的尸检报告目录，以及与当前任务相关的尸检报告。

提炼并保留一份简短的工作检查清单：

- 与当前任务相关的重复失败模式
- 明确禁止的做法
- 编码前必须完成的检查
- 收尾前必须完成的验证

在这份检查清单明确之前，不要开始写代码。

## 编码前

面对任何编码任务，都按这个顺序执行：

1. 用一句话定义本次任务目标。
2. 阅读尸检报告列表。
3. 只挑出与当前任务相关的报告。
4. 从这些报告中提炼本次任务专用检查清单：
   - 哪些假设必须先验证
   - 哪些命令、流程必须使用
   - 哪些反模式必须避开
   - 哪些完成检查必须执行
5. 只有完成以上步骤，才能开始写或改代码。

如果没有找到任何尸检报告，可以正常继续，但要明确说明：这次没有可参考的既往尸检。

## 执行中

每到一个阶段边界，都要重新检查：

- 当前工作是否违反了既有尸检中的规则
- 是否应该尽早纠正，而不是最后打补丁
- 当前结果是否已经验证，能够进入下一阶段

## 阶段完成或收尾时

当一个阶段完成，或者准备结束本次工作时，如果出现以下任一情况，就必须补写一份尸检报告：

- 错误假设
- 无效探索
- 工具使用错误
- 可以避免的重试
- 验证流程缺失或错误
- 任何值得下次避免的问题

如果本次没有真实事故，不要硬写尸检。只有用户明确要求时，才写简短的“无事故收尾说明”。

## 存放位置与命名

默认将尸检报告存放在项目根目录的 `postmortems/` 目录下。

文件命名格式必须为：

`序号-简短标题.md`

规则：

- 序号为三位递增序号，如 `001`、`002`
- 标题使用小写英文和短横线
- 标题应简短描述问题对象与核心错误，不得使用空格、中文或特殊符号

同时维护 `postmortems/README.md` 作为尸检目录索引。在开始编码前，先读
`postmortems/README.md`，再按需读取相关尸检。

## 尸检报告格式

使用以下结构：

```txt
# 标题

**日期**: YYYY-MM-DD
**影响**:
**发现人**:

## 问题
<!-- 发生了什么，影响了什么 -->

## 现象
<!-- 最小复现命令 + 关键报错 -->

## 根因
<!-- 错误假设 / 实际约束 / 缺失检查点 -->

## 修复
<!-- 当时怎么恢复 / 以后正确做法 -->

## 预防
<!-- 可执行规则，不写空话 -->

```

## README格式

保持简洁

```txt
# Postmortems

- [001 · 示例标题](./001-example.md)
- [002 · 示例标题](./002-example.md)
```
